
# Frontend Review: Required Changes and Current Problems

## 1. Overall Design Direction

The current interface does not resemble the original design intent.

The dark UI mock-up looked significantly better and should be used as the main visual reference. The current implementation does not feel like a production-grade application. It lacks visual hierarchy, consistent formatting, and a clear design system.

We need to define a stronger reference for the interface:

* What information should each screen display?
* How should that information be organized?
* Which elements make the reference design effective?
* What should our application look and feel like?
* Which parts can be adapted or reproduced without copying the entire design?

The goal should be a sleek, modern, and operationally useful interface rather than a collection of plain lists and disconnected panels.

---

## 2. Sessions Panel

### Current Problems

The sessions panel currently feels unfinished and difficult to navigate.

Specific issues include:

* Sessions appear as a plain list without meaningful formatting.
* There are no timestamps.
* There is no clear information hierarchy.
* Full file paths are displayed when only the relevant project name may be needed.
* Internal fields such as notes, target notes, or similar metadata are visible even though they are not useful in this view.
* Session states are unclear .
* The interface in-session does not distinguish between an empty session and a session that is still loading.
* Too many controls are displayed directly instead of being grouped.

### Required Changes

Each session should display a concise and readable summary, including:

* A meaningful session name or generated title
* The project being worked on
* The session status
* The creation or last-updated timestamp
* A short indication of the current or most recent task

The session list should have clear visual hierarchy and spacing.
The in-session messages also.

Secondary actions should be moved into a three-dot menu. Only the most important information and actions should remain visible.

The interface also needs clearly defined session states, such as:

* Loading
* Empty
* Active
* Waiting for input
* Completed
* Failed

A loading spinner, skeleton state, or similar indicator should appear while a conversation is being loaded.

## 3. Individual Session View

### Message Formatting

The conversation view lacks structure.

Problems include:

* Weak distinction between messages
* No timestamps
* No clear hierarchy
* Message boxes feel visually outdated
* Poor use of spacing and typography
* No obvious separation between instructions, system events, task updates, and responses.
* No font or rendering is present.

The session view should use modern message formatting with consistent spacing, timestamps, message roles, and clear visual separation between different content types.

### Scrolling and Layout

The message history, instruction input, and keyboard/input area currently interfere with each other.

The message history should have its own independent scroll container. Opening the instruction panel or keyboard should not prevent the user from scrolling to the top or bottom of the conversation.

The layout should be divided into:

1. A fixed session header
2. An independently scrollable message history
3. A fixed or sticky instruction input area

The user should always be able to navigate the full conversation regardless of whether the instruction input is open.

### Instruction Panel

When opening the instruction interface, unrelated information such as keys, credit-card information, location data, or other metadata becomes visible.

This creates unnecessary clutter and may expose information that should not appear in the main workflow.

Only information directly relevant to sending the instruction should be visible.

## 4. Session Navigation and Secondary Panels

The current interface may not need a permanent global side menu. However, some form of session-level navigation may still be necessary.

A session-specific navigation area could provide access to:

* Conversation
* Tasks
* Modified files
* Session metadata
* Logs or activity
* Settings or secondary actions

This could be implemented as tabs, a compact side panel, or buttons inside the session header.

The important requirement is that related information remains attached to the session instead of being distributed across disconnected global screens.

## 5. Files View

The current Files screen is confusing.

It is not clear whether it represents:

* Files in the repository
* Files modified during a session
* Files modified by a specific task
* Files generated by the application

The relationship between files, tasks, and sessions is also unclear.

For example, selecting “Go to session” from a task may open a session that does not show any file changes. This makes it difficult to understand why those files are being displayed or what they are connected to.

### Recommended Direction

First, define the purpose of the Files feature.

If the purpose is to show files modified during a session, the files should appear inside that session.

If the purpose is to show files modified by a task, they should appear inside the task details.

A separate global Files screen is probably unnecessary unless it represents the complete repository or provides a meaningful cross-session file history.

Each file entry should clearly show:

* File name
* Short relative path
* Type of change
* Related task
* Related session
* Modification timestamp

The interface should make the relationship between file, task, and session immediately understandable.

## 6. System Screen

The current System screen mixes unrelated concepts:

* Application state
* Nodes
* Offline nodes
* Live activity
* Task events
* Navigation to sessions

This makes the purpose of the screen unclear.

Fields such as targets/nodes, and offline nodes do not appear operationally useful and should not occupy prominent space.

### Recommended Direction

The System screen should become a clear system status or operational summary.

It could include:

* Current system state
* Running tasks
* Queue status
* Connected services
* Errors or warnings
* Resource usage
* Recent important events

Live activity can remain, but it should support operational understanding rather than act as an unstructured event stream.

The current combination of system summary and live activity feels conflated. These concepts should either be separated or redesigned into a unified operational dashboard.

---

## 7. Tasks and Activity

The Tasks screen should be the main place for understanding what is happening across the application.

The current structure separates tasks into sections such as:

* Needs attention
* Recently completed

This is not working well because every failed task appears under “Needs attention,” turning it into a dump of failures rather than a useful prioritization system.

### Recommended Direction

Tasks should primarily be displayed chronologically, with clear status and hierarchy.

Each task should show:

* Task name
* Related project
* Related session
* Current status
* Start and completion time
* Whether user action is required
* Short result or failure reason

Statuses could include:

* Queued
* Running
* Waiting for input
* Completed
* Failed
* Cancelled

“Needs attention” should only include tasks that genuinely require user action. A failed task should not automatically be considered something that needs attention.

Recent task activity could be presented as a chronological timeline. This would allow the user to understand the actual flow of work and open the relevant session when necessary.

Completed tasks should be contained through pagination, collapsing, filtering, or a limited recent-history view so the screen does not grow indefinitely.

---

## 8. Required Product Principles

The redesigned frontend should follow these principles:

1. Every screen must have a clear purpose.
2. Related information must remain connected.
3. Sessions, tasks, and files must show their relationships.
4. Loading, empty, active, completed, and failed states must be visually distinct.
5. Secondary controls should be moved into menus.
6. Timestamps should be used consistently.
7. Internal metadata should not be shown unless it is operationally useful.
8. The interface should prioritize hierarchy, readability, and navigation.
9. The design should use the dark mock-up as the primary visual reference.
10. The result should feel like a coherent production application, not a collection of development panels.
